Blog reboot, part II

In my previous blog post I already mentioned that I have been hard at work even though it might have been quiet on my blog for some time. One of the things that has kept me busy over the past couple of months is the work that I have done on a new code repository on GitHub. Being just one of the more than 100 million repositories hosted on this site is a bit of a change for me because, up until now, I have hosted all my Subversion repositories myself. I think it is important to point out that the repositories which predate my newly created GitHub repository can still be accessed and, looking back, it is interesting to see the evolution of the code.

In hindsight I would qualify the very first project I blogged about as a collection of Node.js API prototypes. The successor project was primarily aimed at Dockerizing the application’s platform. In the new GitHub repo this is all brought together with major code improvements to boot.

GitHub repository api_hub_rpr contains the code to construct a fully Dockerized software stack to;

  1. make requests to external APIs in response to incoming HTTP(S) requests,
  2. persist the JSON or XML contained in the external API's response and
  3. provide a response to the initial request either from the connected database or by simply passing on the external API's response.

Please refer to the project’s README.md file for an up-to-date listing of the supported APIs & data packets.

At this moment the application has no user interface but, provided the required SSL certificates are installed, a browser can be used to request information from the hub.

I have designed the hub code in such a way that adding additional APIs and/or data packets is easy. I am currently considering the implementation of either the KvK API or LEI API. Make sure to stay tuned for more!
