Blog reboot, part IV

In a previous blog post I mentioned that I moved " … a lot of function code out of classes so as to share it between classes … ". Of course this will only work if the function code is then invoked using the call method with the appropriate "this" reference specified. In my opinion the "this" reference in JavaScript is one of the most challenging concepts of the language. This is especially true in ECMAScript 6 because there is the additional complication that the object the "this" keyword refers to differs between traditional functions and the newly introduced arrow functions.

Luckily a lot of excellent documentation on the topic of "this" is publicly available. For me the "this" reference is, through the concept of an execution context and the execution context stack, closely related to the JavaScript runtime. How the JavaScript runtime functions in a wider environment, for example a browser or Node.js, is perfectly explained in the video below.

Maybe I should just consider it a testimony of JavaScript's immense popularity nowadays but, for me, it is mind-boggling that Philip Roberts was able to garner more than one million views on a video about such an esoteric subject. However that may be, it inspired me to do some JSFiddle experiments;

  1. with a zero delay timeout and
  2. purposely blocking the main thread of a JavaScript application.

Rather than zooming out to focus on the wider environment in which the runtime engine is embedded, Dmitry Soshnikov, in his article JavaScript. The Core (v2), chooses to zoom in & this allows him to treat the runtime's core components in more detail. For me these seven components are the most interesting;

  1. Object (fiddle 1)
  2. Prototype (fiddle 1)
  3. Class (fiddle 2)
  4. Execution context (fiddles 3 & 4)
  5. Environment (fiddles 3 & 4)
  6. Closure (fiddles 5 & 6)
  7. The "this" keyword (fiddles 7 & 8 & 9)

The fiddles mentioned after the individual sections refer to the JavaScript test code I created to help me really understand the article's content. Please be aware that the information density of the article is high and I must confess that, at times, digesting the information was a bit of a struggle. However, at the end of the day, I am glad that I put in the effort and I am convinced that the quality of my code will be better as a result of it.
