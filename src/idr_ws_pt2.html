D&B IDR persistence

This blog post is all about adding persistence to my D&B Direct+ IDentity Resolution (IDR for short) object code. In summary I took three steps to add this functionality to class DplIDR: (1) I created the required database structures, (2) I developed the additional Node.js code and (3), to conclude, I performed a limited number of tests. Below you will find more information about the implementation details of each of these steps.

  1. Database structures. If you read my blog regularly it should come as no surprise that I love the JSONB data type. This data type was first introduced in PostgreSQL version 9.4. For storage purposes both the IDentity Resolution input parameters (field parameters) and the Direct+ end result as returned by the API (field result) are defined as database columns of type JSONB. Column http_stat saves the HTTP status code which is part of the Direct+ REST response. Especially when error conditions occur it can be very helpful to have recorded to this data-element. Column id is a so-called auto increment field. It provides a unique, numeric primary key value for each row of data. Please note, all the SQL statements associated with the required database structures which support my D&B Direct+ object code can be found at the end of the JavaScript file dnb_dpl_objs.js.
  2. Node.js JavaScript object code. From an application logic point of view the IDentity Resolution transaction is persisted in two parts: (1) the input parameters are saved when the IDR REST request is initiated and (2) upon return of the match results the JSON body of the HTTP response is written to the database in conjunction with the HTTP status & a timestamp.
