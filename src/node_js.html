I ❤ Node.js

When building a web application my go-to move is to use C to create an Apache module. In a previous blog post I already explained why Apache is my favourite HTTP server. Versatility is definitely one of Apache's strong points and modules play an important role in achieving this. A number of modules come as part of the server distribution but, if you're slightly more ambitious and know how to program either C or C++, creating your own, customized modules is relatively simple. This blog actually is an Apache module with a bit of client-side JavaScript code bolted on.

Why would I even consider Node.js when I am so enthusiastic about Apache modules? The answer is threefold:

  1. Prototyping. In my experience, when you start work on a C project, it's probably best to do some thinking & designing first. You can only start coding upon conclusion of this phase. This is not my preferred approach when prototyping. Usually, when prototyping, I instantly want to test a couple of high-level concepts. For me the best way to accomplish this is to write & run some bare bones code. Node.js is very helpful in these circumstances because (1) it's extremely easy to get the JavaScript run-time environment ready for play on Ubuntu, (2) JavaScript code is interpreted so there is no need to compile after each code change and (3) the availability of all kinds of high quality libraries allows you to move fast.
  2. Technology. Node.js is a JavaScript run-time environment with an event-driven architecture capable of asynchronous I/O. Everything is aimed at enhancing throughput. I can definitely see the rationale behind this design choice, especially in the case of web applications. When I first started writing Node.js code I was surprised at how much, as a developer, you are forced into this asynchronous way of thinking. This might sound negative but it is, in practice, very helpful. It all makes perfect sense once you get the hang of it.
  3. JavaScript. One of my claims to fame is that I've read the Rhino book cover to cover. Initially my goal was to make my web application's client interface more interactive. At the time I already concluded that JavaScript has a surprising depth to it. A whole new dimension is added to this assessment by being able to use this programming language on the server as well.

I don't won't to give away too much about what comes next but, as a teaser, I can already let you know that, even though I had high expectations when I set out on my journey to implement D&B Direct+ using Linux(/Ubuntu), Node.js and PostgreSQL, this combination performed, in more than one way, even better than I had expected.
